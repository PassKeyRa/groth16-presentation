Toolbox 1 / 4 — finite fields

All arithmetic happens on a clock

𝔽17 — integers mod a prime · 12 + 8 ≡ 3

  • Work modulo a prime p. Numbers wrap around; there is no “bigger than”.
  • Every non-zero element has an inverse — so division always works: 5−1 mod 17 = 7, because 5 · 7 = 35 ≡ 1.
  • Why we care: every wire, every constraint, every polynomial in a SNARK lives in 𝔽p. No floats, no rounding, exact algebra.

Toolbox 2 / 4 — elliptic curves

Point addition is a geometric rule

CHORD RULE: P + Q
  • draw the line through P and Q
  • it hits the curve at a third point
  • reflect over the x-axis → P + Q
DOUBLING: 2P

Same rule with the tangent line at P.

This rule is associative and has an identity 𝒪 — the points form a group. That’s the only fact we need.

Toolbox 2 / 4 — elliptic curves, for real

Over 𝔽17 the “curve” is a cloud of points

y² = x³ + 3 (mod 17) — same algebra, same addition formulas

field order p = 17
where the coordinates live
curve order n = 18
how many points exist — a different number! (17 affine + 𝒪)

Example from the table: (1, 15) + (4, 4) = (16, 6) — computed with the same chord formula, mod 17.
On BN254 both numbers are ≈ 2254: coordinates mod p, but circuit arithmetic mod the curve order r.

Scalar multiplication hides a number

Toolbox 3 / 4 — what hidden values allow

Hidden values: add yes, multiply no

✓ [a] + [b] = [a + b] add two hidden values — point addition
✓ c · [a] = [c·a] scale by a public constant — so any public linear combination works: Σ cᵢ[aᵢ] = [Σ cᵢaᵢ]
✗ [a] · [b] = ? multiply two hidden values — impossible with curve operations alone

A circuit is full of multiplications… we need at least one multiplication check. Enter pairings.

Toolbox 4 / 4 — bilinear pairings

A pairing checks one multiplication

the only property we use — bilinearity: scalars escape into the exponent and multiply

THE MULTIPLICATION CHECKER

e([a]₁, [b]₂) = e([c]₁, [1]₂)

holds  ⇔  a·b = c — verified without ever learning a, b, or c.

ON ETHEREUM · BN254

precompiles 0x06 ecAdd · 0x07 ecMul · 0x08 ecPairing

Internals (Miller loops, tower fields) — out of scope. Treat e as a trusted gadget.

Toolbox 4 / 4 — the concrete instantiation (FYI)

BN254 (“BN128”): three groups, three fields

G₁ — the base curve

y² = x³ + 3  over 𝔽p

coordinates: two numbers mod p
(p = a 254-bit prime)
a point ≈ 64 bytes

G₂ — the twist

y² = x³ + 3∕(9+u)  over 𝔽p²

same story, bigger coordinates: each is a pair a + b·u — like complex numbers mod p
a point ≈ 128 bytes

G_T — not a curve at all

subgroup of 𝔽p¹²×

pairing outputs live in a huge extension field (12 numbers mod p) and are multiplied, never sent on-chain

The pairing e : G₁ × G₂ → G_T doesn’t “move” a point between curves — it consumes one point from each group and outputs an 𝔽p¹² element, with the scalars multiplying in the exponent. All three groups share the same prime order r ≈ 2²⁵⁴ — so all scalars (witness values, circuit arithmetic) live in 𝔽r, while point coordinates live in 𝔽p, 𝔽p², 𝔽p¹². Two different fields — the slide-5 distinction, at full scale.

Arithmetization — step 1

From claim to circuit

THE CLAIM

“I know x such that
x³ + x + 5 = 35”

(the secret: x = 3)

FLATTENED — ONE GATE PER LINE
gate 1:  v₁ = x · x
gate 2:  v₂ = v₁ · x
gate 3:  out = v₂ + x + 5
  • Each gate: at most one multiplication
  • Additions and constants ride along inside a gate’s linear combinations — they’re free

Rank-1 Constraint System — checked by hand

The witness satisfies every row

gate 1:  (A₁·w)(B₁·w) = 3 · 3  = 9  = C₁·w  ✓ // x·x = v₁
gate 2:  (A₂·w)(B₂·w) = 9 · 3  = 27 = C₂·w  ✓ // v₁·x = v₂
gate 3:  (A₃·w)(B₃·w) = 35 · 1 = 35 = C₃·w  ✓ // v₂+x+5 = out

satisfying witness  ⇔  correct computation

THE PROBLEM

Checking all rows = re-running the whole program. Real circuits have millions of rows. We need to compress m checks into one.

The key trick

Why polynomials? One point tells all

two different degree-d polynomials can agree on at most d points (here d = 3)

THE WHOLE POINT OF POLYNOMIALS

Comparing two vectors of m entries = m comparisons, one per entry.
Comparing two polynomials = 1 comparison, at a single random point.

WHY THAT’S SOUND — SCHWARTZ–ZIPPEL

If they differ anywhere, they differ at a random τ from a ≈ 2²⁵⁴-size field with overwhelming probability — cheating survives with chance d / p ≈ 2⁻²³⁰.

so: encode all m constraints
as polynomials → check once, at τ

The encoder

Lagrange interpolation: vectors become polynomials

vector (4, 7, 1) at x = 1, 2, 3 → the unique parabola through the three points

This is the vector → polynomial converter. Treat entry i as the value at x = i, interpolate — every vector becomes a polynomial, losslessly, and back.

  • Each faint “bump” ℓᵢ(x) equals 1 at its point, 0 at the others
  • Scale each bump by its value and add — the sum passes through every point
  • n values ⇔ unique polynomial of degree < n — so our R1CS vectors qualify for the one-point equality check

That’s all we need — closed formulas are a footnote.

Quadratic Arithmetic Program

Interpolate the matrices, column by column

each column of A (one witness slot) read down the gates → one polynomial Aᵢ(x) · same for B and C
then bundle with the witness: A(x) = Σ wᵢ·Aᵢ(x) — matrices are gone, polynomials remain

Quadratic Arithmetic Program — a wrinkle

A·B and C agree only at the gates

our real example: A(x)·B(x) has degree 4, C(x) degree 2 —
different curves that cross exactly at the gate points x = 1, 2, 3

SO WE CAN’T JUST WRITE A·B = C

deg(A·B) = 2(n−1), deg(C) = n−1 — as polynomials they can never be equal, even when every gate is satisfied.

FIX — A BALANCING TERM THAT’S 0 AT EVERY GATE

Add something to C’s side that absorbs the degree gap but vanishes at x = 1 … n, so it can’t disturb the gate checks. A polynomial with roots at all gates is by definition a multiple of Z(x) = (x−1)(x−2)⋯(x−n) — so the term must have the form H(x)·Z(x). The prover can’t smuggle in anything else.

Quadratic Arithmetic Program

All gates become one divisibility

A(x)·B(x) − C(x) for our real example — it touches zero exactly at gates x = 1, 2, 3

A(x)·B(x) = C(x) + H(x)·Z(x)

Z(x) = (x−1)(x−2)(x−3) — the vanishing polynomial · the balancer from the previous slide, written on C’s side · equivalently: A·B − C = H·Z

  • gate i satisfied ⇔ the difference A·B − C is zero at x = i (plot ←)
  • zero at all gates ⇔ Z(x) divides it — the quotient H(x) = (A·B − C)/Z exists with no remainder
  • prover’s job: compute and commit to H(x). One divisibility = the whole execution.

α, β — force honest A, B, C

ATTACK ON THE NAIVE SCHEME

Verifier only sees three opaque points and a pairing equation. Nothing forces them to come from real polynomials — a cheater can pick arbitrary points that balance the equation, witness-free.

FIX — SECRET SHIFTS FROM THE SETUP

A = [α + A(τ) + …]₁
B = [β + B(τ) + …]₂

The check now requires the cross-terms around α·β to cancel — only possible if A, B, C were built from the same witness. Faking it needs α, β themselves; the world only has [α]₁, [β]₂.

Greek letters, one attack at a time

γ, δ — wall between public and private

REMAINING ATTACK

Public inputs sit in a verifier-computed sum; private witness sits in C. Both are linear combinations of similar setup points — a prover could slide terms from one side into the other and fake a public input.

FIX — DIVIDE THE LANES BY DIFFERENT SECRETS

Public-input points are published divided by γ; witness & quotient points divided by δ. The pairing check multiplies them back against [γ]₂ and [δ]₂ — a term in the wrong lane keeps a stray 1/γ or 1/δ and can never cancel.

γ
job: seals the public-input lane (verifier’s side)
δ
job: seals the witness + H·Z lane (prover’s side)

The last ingredient — the “zk” in zk-SNARK

Salt r, s: same witness, fresh proof every time

WITHOUT SALT — DETERMINISTIC PROOFS

Same witness → byte-identical proof. Anyone who can guess your witness (small search space: an address, a small amount, a 4-digit PIN) just runs the prover and compares bytes. Guess confirmed — zero knowledge broken.

WITH SALT

The prover samples fresh random r, s for every proof — (A, B, C) become uniformly random points that still satisfy the equation. Two proofs of the same witness look completely unrelated.

THE SALTED PROOF

A = [α + A(τ) + r·δ]₁
B = [β + B(τ) + s·δ]₂
C = [ ⟨private Ψ⟩∕δ + H·Z∕δ + s·A + r·B − r·s·δ ]₁

the rose terms in C are built to exactly cancel the r·δ and s·δ cross-terms when the product A·B expands — the equation still balances, for any r, s
· δ-scaling keeps the salt inside the prover’s lane from the last slide

Auditor’s corner

Where Groth16 systems actually break

1

Under-constrained circuits

The R1CS admits witnesses the developer never intended — the #1 real-world bug class. The proof system is fine; the encoding lies.

2

Toxic waste survives

Setup secrets not destroyed → unlimited forgery. Zcash’s 2018 counterfeiting vulnerability lived in ceremony artifacts.

3

Input aliasing

Public inputs are field elements mod r — accepting values ≥ r gives two integer encodings of the same input (nullifier double-spend pattern).

4

Proof malleability & point checks

(A, B) ↦ (−A, −B) still verifies — don’t use the proof itself as a uniqueness key. Verifier must check points are on-curve / in-subgroup.
